Ось відповіді на запитання:

1. Що таке Dockerfile?

Dockerfile — це текстовий файл зі сценарієм інструкцій, який використовується для автоматизації створення образу Docker. У цьому файлі описано всі кроки для створення образу, включно з установкою залежностей, копіюванням файлів і налаштуванням контейнера.

Приклад простого Dockerfile:

dockerfile
FROM python:3.9-slim
WORKDIR /app
COPY . /app
RUN pip install -r requirements.txt
CMD ["python", "app.py"]

2. Як дізнатися IP-адресу контейнера?

Для отримання IP-адреси запущеного контейнера використовується команда:

bash
docker inspect -f '{{range.NetworkSettings.Networks}}{{.IPAddress}}{{end}}' <container_id>

Або детальна інформація про мережу:

bash
docker inspect <container_id>

3. Які команди є в Docker?

Найпопулярніші команди Docker:

  • Запуск контейнера:
    bash
    docker run <image_name>
  • Список контейнерів:
    bash
    docker ps -a
  • Створення образу з Dockerfile:
    bash
    docker build -t <image_name> .
  • Видалення контейнера:
    bash
    docker rm <container_id>
  • Перехід усередину контейнера:
    bash
    docker exec -it <container_id> /bin/bash

4. Як взаємодіють два контейнери?

  1. За допомогою мережі Docker. Якщо контейнери підключені до однієї мережі, вони можуть взаємодіяти за іменами хостів. Наприклад:

    bash
    docker network create my_network
docker run --network=my_network --name=container1 ...
docker run --network=my_network --name=container2 ...

    У цьому випадку container1 може звертатися до container2 за адресою http://container2.

  2. Обмін даними через томи (Volumes). Том дозволяє контейнерам зберігати та обмінюватися файлами:

    bash
    docker run -v shared_data:/data ...

Docker образ (Docker image) — це шаблон для створення контейнерів. Він містить усі необхідні файли, залежності, конфігурації та інструкції, які дозволяють запустити програму в контейнері.

Головні характеристики Docker образу:

  1. Шарова структура (layers):

    • Docker образ складається з шарів, які створюються під час виконання інструкцій у Dockerfile. Наприклад, команда RUN додає новий шар.
    • Шари оптимізують використання пам'яті, оскільки один і той самий шар може використовуватися різними контейнерами.

  2. Базовий образ:

    • Образи можуть базуватися на інших образах. Наприклад, python:3.9-slim є базовим образом, на основі якого можна створити кастомний образ для Python-додатків.

  3. Неможливість змін:

    • Образи є статичними, тобто після створення вони не змінюються. Зміни вносяться у контейнері, який працює на основі образу.

  4. Зберігання в реєстрах:

    • Образи зберігаються у Docker Registry (наприклад, Docker Hub), що дозволяє легко ділитися ними або завантажувати готові образи.

Як працює Docker образ?

  1. Створення образу:

    • Створюється за допомогою Dockerfile:
      bash
      docker build -t my_image .
    • У результаті отримуємо образ із заданою конфігурацією.

  2. Запуск контейнера:

    • На основі образу створюється контейнер:
      bash
      docker run my_image

  3. Оптимізація:

    • Образ використовує шари з кеша, якщо вони вже створені, що пришвидшує роботу.

Приклад базового образу:

Образ Python:

dockerfile
FROM python:3.9-slim
RUN pip install flask

Цей образ включає базовий образ python:3.9-slim та додає Flask.

Команди для роботи з образами:

  • Список локальних образів:
    bash
    docker images
  • Видалення образу:
    bash
    docker rmi <image_id>
  • Отримання образу з Docker Hub:
    bash
    docker pull nginx
  • Перегляд деталей образу:
    bash
    docker inspect <image_id>

Підсумок:

Docker образ — це основа для створення контейнерів. Це як "шаблон", який містить усе необхідне для запуску програми в ізольованому середовищі.

У Docker є кілька способів переглядати та керувати логами контейнерів. Логи — це вихідна інформація (stdout і stderr), яка генерується запущеними контейнерами.

Як переглядати логи контейнера?

  1. Перегляд логів контейнера:

    bash
    docker logs <container_id_or_name>
    • Ця команда показує весь вихід логів із моменту запуску контейнера.

  2. Показ логів у реальному часі: Для "живого" перегляду логів використовуйте опцію -f (схоже на команду tail -f):

    bash
    docker logs -f <container_id_or_name>
    • Це зручно, якщо потрібно стежити за подіями в реальному часі.

  3. Обмеження кількості логів: Для перегляду лише останніх логів використовуйте опцію --tail:

    bash
    docker logs --tail 50 <container_id_or_name>
    • Наприклад, --tail 50 показує лише останні 50 рядків.

  4. Логи із часовими мітками: Для додавання часових позначок до логів:

    bash
    docker logs --timestamps <container_id_or_name>

Як працювати з великими логами?

Якщо логів дуже багато, їх можна фільтрувати або перенаправляти в інший файл:

bash
docker logs <container_id_or_name> > logs.txt

Також можна використовувати сторонні утиліти, наприклад grep:

bash
docker logs <container_id_or_name> | grep "error"

Налаштування ротації логів

За замовчуванням Docker зберігає всі логи, що може призводити до переповнення диска. Щоб обмежити розмір логів, використовуйте драйвери логів:

  1. Налаштування в daemon.json: У файлі /etc/docker/daemon.json додайте:

    json
    {
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
    • max-size: максимальний розмір файлу логів.
    • max-file: кількість файлів для ротації.

  2. Перезапустіть Docker:

    bash
    systemctl restart docker

  3. Або для окремого контейнера: Під час запуску контейнера:

    bash
    docker run --log-driver json-file --log-opt max-size=10m --log-opt max-file=3 <image>

Драйвери логів Docker

Docker підтримує різні драйвери логів для зберігання або передачі логів:

  • json-file (за замовчуванням)
  • syslog (надсилає логи до системного логгера)
  • journald (для систем із systemd)
  • fluentd, awslogs, gelf, splunk (для інтеграції з лог-системами)

Для зміни драйвера логів:

bash
docker run --log-driver=syslog <image>

Перевірка журналу помилок Docker

Якщо є проблеми із самим Docker (а не контейнером), перевірте журнал сервісу:

bash
journalctl -u docker.service

Приклади

  1. Переглянути останні 100 рядків логів контейнера:

    bash
    docker logs --tail 100 <container_id>

  2. Перегляд логів у реальному часі із фільтрацією:

    bash
    docker logs -f <container_id> | grep "warning"

  3. Перегляд логів зі збереженням у файл:

    bash
    docker logs <container_id> > /path/to/logs.txt

Логи є ключовим інструментом для діагностики проблем із контейнерами, тому важливо знати, як з ними працювати.

Docker label (лейбл) — це метадані, які додаються до образів, контейнерів, або інших об'єктів Docker. Вони використовуються для опису об'єкта або його властивостей. Лейбли складаються з ключів і значень, які можуть зберігати будь-яку інформацію, наприклад, про автора образу, версію, середовище тощо.

Навіщо потрібні Docker-лейбли?

  1. Організація та ідентифікація: Наприклад, додавання автора або інформації про версію.
  2. Фільтрація: Лейбли можна використовувати для пошуку контейнерів або образів.
  3. Автоматизація: Інструменти, такі як Kubernetes, Docker Compose і CI/CD системи, можуть використовувати лейбли для автоматизації.
  4. Документація: Лейбли допомагають надавати додаткову інформацію про об'єкт.

Як додати лейбли?

1. У Dockerfile:

Лейбли додаються за допомогою інструкції LABEL:

dockerfile
LABEL <key>=<value>

Приклад:

dockerfile
LABEL maintainer="developer@example.com"
LABEL version="1.0"
LABEL description="Це демо-образ для мого застосунку"

У результаті ці лейбли стануть частиною створеного образу.

2. Під час запуску контейнера:

Лейбли можна додавати до контейнера під час запуску:

bash
docker run --label <key>=<value> <image_name>

Приклад:

bash
docker run --label env=production my_image

Як переглянути лейбли?

  1. Перегляд лейблів образу:

    bash
    docker image inspect <image_name> --format='{{json .Config.Labels}}'
    • Відобразить усі лейбли, додані до образу.

  2. Перегляд лейблів контейнера:

    bash
    docker inspect <container_id_or_name> --format='{{json .Config.Labels}}'

Як використовувати лейбли для фільтрації?

За допомогою лейблів можна відфільтрувати об'єкти Docker.

  1. Фільтрування контейнерів:

    bash
    docker ps --filter "label=<key>=<value>"

    Приклад:

    bash
    docker ps --filter "label=env=production"

  2. Фільтрування образів:

    bash
    docker images --filter "label=<key>=<value>"

Як видалити лейбли?

Лейбли не можна змінювати або видаляти в існуючому образі чи контейнері. Є два підходи:

  1. Для образів: Потрібно створити новий образ без небажаних лейблів.
  2. Для контейнерів: Зупинити та запустити новий контейнер без вказаних лейблів.

Приклади використання

  1. Додавання інформації про версію та автора:

    dockerfile
    LABEL version="2.0" maintainer="admin@example.com"

  2. Фільтрування запущених контейнерів за середовищем:

    bash
    docker ps --filter "label=env=staging"

  3. Додавання лейблів у docker-compose.yml:

    yaml
    services:
  app:
    image: my_image
    labels:
      env: production
      version: 1.0

Переваги лейблів:

  • Легкий спосіб додати структуровані метадані.
  • Спрощення керування контейнерами, образами й іншими ресурсами.
  • Інтеграція з інструментами оркестрації, такими як Kubernetes або Docker Swarm.

cmd and run
Програма ChatGPT сказала:

У Dockerfile інструкції CMD і RUN використовуються для різних цілей. Ось їх основна різниця та як їх використовувати:

1. CMD (Command)

  • Використовується для вказівки команди, яка виконується під час запуску контейнера.
  • Це, по суті, "точка входу" контейнера (entrypoint), яка виконується лише під час старту.
  • Ключова особливість: у Dockerfile може бути тільки одна інструкція CMD, і якщо вказати кілька, враховується остання.

Формати використання:

  1. Exec-формат (рекомендований):

    dockerfile
    CMD ["executable", "param1", "param2"]

    Приклад:

    dockerfile
    CMD ["python", "app.py"]

    Це форматує команду в масив, забезпечуючи коректне виконання.

  2. Shell-формат:

    dockerfile
    CMD command param1 param2

    Приклад:

    dockerfile
    CMD python app.py

    У цьому випадку команда виконується через /bin/sh -c.

  3. Параметри для іншої команди (разом із ENTRYPOINT):

    dockerfile
    ENTRYPOINT ["python"]
CMD ["app.py"]

    У цьому випадку CMD передає параметри команді, вказаній в ENTRYPOINT.

Що станеться, якщо CMD не вказано?

Якщо в Dockerfile немає інструкції CMD, використовується значення з базового образу (наприклад, з інструкції FROM).

2. RUN

  • Використовується для виконання команд під час створення образу (на етапі build).
  • Зазвичай застосовується для встановлення пакетів, копіювання файлів тощо.
  • Інструкція RUN додає новий шар до образу (кожна команда RUN створює окремий шар у файловій системі образу).

Формати використання:

  1. Exec-формат:

    dockerfile
    RUN ["executable", "param1", "param2"]

    Приклад:

    dockerfile
    RUN ["apt-get", "install", "-y", "curl"]

  2. Shell-формат:

    dockerfile
    RUN command param1 param2

    Приклад:

    dockerfile
    RUN apt-get update && apt-get install -y curl

    У цьому випадку команда виконується через /bin/sh -c.

Основні відмінності між CMD та RUN

Приклад Dockerfile із CMD та RUN:

dockerfile
# Базовий образ
FROM python:3.9-slim

# Виконання команд на етапі build
RUN apt-get update && apt-get install -y curl
RUN pip install flask

# Копіювання файлів
COPY app.py /app/

# Вказуємо робочу директорію
WORKDIR /app

# Вказуємо команду для запуску контейнера
CMD ["python", "app.py"]

Додаткова інформація:

  • CMD часто використовується для запуску основної програми вашого контейнера.
  • RUN застосовується для підготовки середовища (установлення залежностей, налаштування тощо).
